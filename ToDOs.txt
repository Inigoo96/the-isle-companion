# Proyecto: The Isle Companion — backend multi-tenant (sesión inicial)

## Contexto
The Isle Companion es una app de escritorio (overlay) para The Isle, rama Evrima, que da
info al jugador sin tocar el juego (solo lee el portapapeles para las coordenadas). Ya
existe una versión funcional en Electron + JavaScript vanilla con: overlay transparente
con click-through, mapa con posición vía clipboard, stats por etapa, mutation builder,
growth timer y prime list. Los datos viven en JSON locales (dinos.json, mutations.json,
zones.json).

Nueva dirección: convertirlo en una plataforma multi-tenant para que cada servidor de
The Isle pueda "integrar" la app y esta se adapte a su configuración. La pieza nueva y
central es un backend.

## Arquitectura objetivo
- Overlay (cliente Electron, ya existe) -> Backend (Spring Boot + PostgreSQL, NUEVO, el
  intermediario) -> opcionalmente los servidores del juego vía RCON (solo lectura, lado
  servidor).
- Panel de admin web (React + Vite, NUEVO, más adelante) para que los dueños de servers
  se registren y configuren su server.
- Identidad de jugador mediante login con Steam (Steam OpenID).
- Por qué hace falta el backend intermediario: las credenciales RCON dan control total
  del servidor (banear, kickear, etc.) y NO pueden ir en una app que se reparte a los
  jugadores. El backend las guarda cifradas en el lado servidor y al overlay solo le
  expone datos públicos.

## Límites (definen el alcance — respétalos)
- La app NO gestiona ni replica el "garage" de dinos ni los comandos propios de cada
  server: eso es del bot de Discord de cada servidor. La app NUNCA escribe en el juego.
- La app posee: el overlay + un tracker personal del jugador (su prime list, prefs) +
  datos globales del juego (catálogo) + config de SOLO LECTURA por servidor.
- NO guardamos los dinos del jugador (cada server ya los guarda). Si hace falta el dino
  actual, se lee EN VIVO por RCON; no se persiste.
- NO parsear Game.ini: el formato cambia entre ramas/versiones/hosts. La config por
  server la introduce el admin en el panel.

## Dos niveles (orden de construcción)
- Nivel 1 (PRIMERO, sin RCON, sin secretos): datos globales del juego + config que el
  admin mete a mano + tracker personal del jugador.
- Nivel 2 (después, solo servers integrados): datos en vivo por RCON. getplayerdata
  devuelve, por jugador spawneado, su Steam/EOS ID, el dino actual (DinoClass), Growth %,
  Health/Stamina/Hunger/Thirst % y Location. Se cruza por el Steam64 del jugador.

## Stack
- Backend: Spring Boot (Maven) con Spring Web, Spring Data JPA, Spring Security,
  Validation, Flyway y driver de PostgreSQL. Arquitectura por capas (controllers ->
  services -> repositories). La API devuelve DTOs, NUNCA expone entidades JPA.
- Base de datos: PostgreSQL. El esquema viene en el archivo adjunto
  V1__initial_schema.sql (migración Flyway). Respétalo: usa Flyway con
  spring.jpa.hibernate.ddl-auto=validate (no generes el esquema con Hibernate).
- Panel admin: React + Vite (más adelante).
- Overlay: sigue en Electron + JS vanilla (ya existe, NO reescribir).
- Coste: proyecto personal, prioriza gratis/barato. Nada de AWS por defecto; el deploy
  futuro será VPS barato o tiers gratuitos.

## Notas de RCON (contexto para el Nivel 2)
- Evrima usa un protocolo RCON binario propio (no Source RCON; mcrcon no sirve).
- getplayerdata = opcode 0x77.
- No existen comandos de spawn/teleport/heal.
- Las credenciales RCON van cifradas en la BD, lado servidor, NUNCA en el cliente.

## Modelo de datos
V1__initial_schema.sql (migración Flyway): enums para conjuntos cerrados; contenido
global (dinos, dino_stats, dino_growth_stages, mutations, dino_mutations, prime_tasks);
identidad y tenancy (accounts por steam_id varchar, servers, server_allowed_dinos);
progreso personal (prime_runs, prime_run_tasks). steam_id es varchar (17 dígitos Steam64
no caben en entero JS). Dos enums de etapa DISTINTOS: stat_stage vs growth_stage.

V2__add_zones.sql (AÑADIR): tabla zones con enums zone_category (patrol/migration/
sanctuary) y zone_shape (circle/polygon). Geometría como jsonb:
  - circle: {center:[x,y], radius:n}
  - polygon: {points:[[x,y],...]}
Sin PostGIS (coordenadas del juego, no geo real). UNIQUE(name, category).
